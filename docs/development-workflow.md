# 需求 / Bug 处理工作流

这份文档定义本项目后续默认采用的标准工作流，目的是让每一次需求实现和 bug 修复都遵循同一套节奏，避免：

- 旧服务未停止导致环境污染
- 测试残留影响正式环境
- 用户验收时拿到的是未重启或未清理的服务
- iTerm 会话被创建后没有正确清理

## 标准流程

### 1. 用户提出需求或 Bug

进入开发前，至少要明确以下信息中的一部分：

- 复现步骤
- 预期结果
- 实际结果
- 页面 URL
- 截图
- 报错文案
- 相关元素定位

如果信息不完整，先补充最小复现条件，再进入改代码阶段。

### 2. 修改代码

根据需求定位对应文件并修改实现。

建议同步完成：

- 相关样式或脚本调整
- 文档更新
- 必要的默认值修正
- 生命周期或资源清理逻辑修正

### 3. 执行 `./stop.sh`

在正式进入测试前，先停止当前服务并清理旧环境：

```bash
./stop.sh
```

目的：

- 停掉旧的 FastAPI 服务
- 清理项目托管的 iTerm 会话
- 避免旧状态干扰测试结果

### 4. 启动测试工程，并自行测试后再关闭

这一阶段由开发者自己完成，不直接交给用户。

建议动作：

1. 启动测试用服务或页面
2. 复现原始问题
3. 验证修复结果
4. 回归关键流程
5. 关闭测试过程产生的临时环境

这一阶段的原则是：

- 测试环境不留脏数据
- 测试窗口不留给用户验收
- 测试结果以开发者自证为主

### 5. 再次执行 `./stop.sh`

如果测试过程中重新拉起了服务、生成了测试终端或临时会话，在交付前要再次清理：

```bash
./stop.sh
```

目的：

- 清掉测试阶段产生的 iTerm 会话
- 确保正式交付前环境是干净的

### 6. 使用 `./start.sh` 启动正式服务

确认测试通过后，再启动一份正式用于用户验收的服务：

```bash
./start.sh
```

目标：

- 用户打开页面就能直接验收
- 页面拿到的是最新代码
- 服务状态可用且稳定

### 7. 通知用户并交付验收

最后回复用户时，需要明确：

- 改了什么
- 影响了哪些文件
- 当前服务已重新启动
- 用户如何复测
- 如果有已知限制，也要明确说明

推荐交付口径：

- 已完成修改
- 已完成自测
- 已使用 `./start.sh` 启动正式服务
- 可直接打开页面验收

## 推荐执行模板

以后默认按下面顺序执行：

1. 用户提出需求 / bug
2. 修改代码
3. `./stop.sh`
4. 自测（启动、测试、关闭）
5. `./stop.sh`
6. `./start.sh`
7. 通知用户验收

## 为什么必须这样做

这是本项目当前最稳妥的做法，因为项目具备以下特点：

- 会真实创建 iTerm 会话
- 会和本地 iTerm Python API 建立长连接
- 会保留监控状态与 WebSocket 通道
- 旧环境和测试环境很容易污染用户验收环境

统一流程的价值在于：

- 降低环境污染
- 降低误判 bug 的概率
- 避免 iTerm 会话残留
- 保证每次交付给用户时都是“可直接验证”的状态

## 脚本说明

### `./start.sh`

负责：

- 停掉旧服务
- 启动新服务
- 写入 pid 文件
- 等待健康检查成功

### `./stop.sh`

负责：

- 停掉服务
- 清理本项目托管的 iTerm 会话
- 尝试退出空闲 iTerm

## 适用范围

这条工作流适用于：

- 功能开发
- bug 修复
- 交互调整
- 页面样式修改
- iTerm 生命周期问题处理
- WebSocket / FastAPI / 前端联调问题
