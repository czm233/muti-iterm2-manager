# 多 iTerm2 管理器：首版实现路线

## 目标定义

我们要做的不是“另一个终端模拟器”，而是一个 **本地 macOS 管理系统**：

- 批量创建、关闭并控制多个 iTerm2 窗口
- 用 Web 看板展示这些窗口的状态
- 允许用户从看板一键放大、聚焦、最小化某个 iTerm2 窗口
- 在后台持续检测任务完成状态，并把状态同步回看板

## 推荐总体架构

### 1. `controller-daemon`（建议 Python）

负责和 iTerm2 官方 API 打交道：

- 创建窗口 / tab / session
- 发送文本命令
- 读取当前输出
- 订阅焦点或其他通知
- 维护 `windowId / tabId / sessionId / taskId` 映射关系

推荐原因：iTerm2 官方自动化能力以 Python API 最自然。

### 2. `window-bridge`（建议先用 Hammerspoon）

负责 macOS 层面的原生窗口控制：

- 设置窗口位置和大小
- 切换焦点、置顶
- 最小化 / 恢复
- 必要时发送物理按键，如 `Ctrl+C`

第一版建议把它做成一个独立桥接层，而不是塞进 Web 服务里。

### 3. `web-backend`（建议 FastAPI + WebSocket）

负责对前端提供：

- 终端列表
- 布局状态
- 任务状态
- 操作 API
- 实时事件推送

### 4. `web-frontend`（建议 React / Vite）

负责做看板：

- 每个 iTerm2 窗口对应一个矩形卡片
- 显示名称、任务、状态、最近输出摘要
- 点击卡片后调用后端，让对应 iTerm2 窗口放大并获得焦点
- 状态完成后改颜色、打标记、提示用户处理

## 需求映射到实现模块

### 1.1 新建 / 关闭 iTerm2

- 主实现：`controller-daemon`
- 参考：`iTerm2` 官方 Python API、`it2`

### 1.2 控制 iTerm2 大小 / 位置

- 主实现：`window-bridge`
- 参考：`Hammerspoon`、`Rectangle`

### 1.3 输入命令 / `Ctrl+C`

- 文本发送：优先用 iTerm2 `send_text`
- 特殊按键：优先用 `window-bridge` 做真实按键注入

### 1.4 四宫格布局

- 主实现：后端输出布局模板，桥接层执行
- 参考：`Rectangle`、`itermocil`

### 2.1 Web 看板

- 主实现：前后端配合
- 参考：`pyxtermjs` 的 WebSocket + 前端终端通信模型

### 2.2 点击矩形后放大、聚焦、置顶

- 主实现：前端点击事件 -> 后端 API -> `window-bridge`

### 2.3 最小化

- 主实现：`window-bridge`

### 3.1 完成态检测

建议双通道：

- 通道 A：iTerm2 输出轮询 / 流式订阅，做关键词判断
- 通道 B：Claude Code hooks 主动回调本地 webhook

### 3.2 看板变色提醒

- 主实现：后端事件总线 + 前端状态机

## 首版建议的技术取舍

- **先做“状态板 + 原生窗口控制”**，不要一开始就追求浏览器内完整终端镜像。
- **先做单机版**，所有组件都跑在本机，避免一开始引入远程代理复杂度。
- **先做固定布局模板**，例如 `1x2`、`2x2`、`1+3`，不要先做可视化拖拽布局编辑器。
- **先做关键词检测**，Claude hooks 作为增强项接入。

## 推荐里程碑

### M0：底层 POC

- 能创建 4 个 iTerm2 窗口
- 能把 4 个窗口排成四宫格
- 能向指定窗口发送命令
- 能对指定窗口执行聚焦、最小化、关闭

### M1：最小可用看板

- Web 页面展示所有窗口卡片
- 卡片显示状态、标题、最近输出摘要
- 点击卡片可以聚焦对应 iTerm2 窗口

### M2：任务状态识别

- 后台轮询输出
- 识别“完成 / 完毕 / error / waiting for input”等状态
- 看板实时刷新样式

### M3：交互增强

- 支持一键四宫格重排
- 支持一键发送预设命令
- 支持“恢复到工作位 / 临时放大 / 再最小化”流程

### M4：稳定性与可运维性

- 持久化窗口与任务映射
- 崩溃恢复
- 权限检测与启动自检
- 结构化日志

## 关键风险

- iTerm2 官方 API 擅长控制终端会话，但不一定覆盖全部 macOS 窗口几何操作。
- 原生窗口控制通常依赖辅助功能权限，这是第一类系统权限风险。
- “像人在键盘输入一样”的需求，文本注入和物理按键注入要区分设计。
- 如果后面要做真正的“窗口实时缩略图”，复杂度会明显高于“状态板 + 输出摘要”。

## 当前建议

下一步最值得先做的是：

1. 先验证 **Python iTerm2 API + Hammerspoon** 这条组合链是否能覆盖 `创建 / 发命令 / 聚焦 / 四宫格 / 最小化`
2. 验证通过后，再搭一个最薄的 `FastAPI + WebSocket + React` 看板骨架
3. 最后再接入完成态检测和 hooks
